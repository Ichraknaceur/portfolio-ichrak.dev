CareerSignal

An AI-powered job search operating system for technical candidates — combining content generation, editorial scheduling, LinkedIn networking, job tracking, and technical watch into a single agentic platform.

🎯 What it does

CareerSignal is built around one idea: turn professional visibility into career opportunities. For an AI/Data candidate, that means consistently producing quality content, building a targeted network, and tracking every opportunity — all with minimal manual effort.

The platform links four loops together: create professional signal → turn that signal into better conversations → track opportunities and follow-ups → feed everything with structured technical watch.

⚙️ Core Engines

✍️ Content Engine
  • Generate Medium articles and LinkedIn posts from a topic, URL, PDF, or GitHub repo
  • QA loop with critique-refine before publication
  • Multi-provider LLM support (Claude, OpenAI)
📅 Publishing Engine
  • LinkedIn content calendar with editable date, time, content and hashtags
  • Approve / reject / re-open workflow
  • Time-aware autopublish worker
🤝 Networking Engine
  • Search LinkedIn profiles and recruiter targets
  • Profile enrichment via scraping
  • Generate personalized connection notes
  • Outreach status tracking with rate limits
💼 Jobs Engine
  • Search and deduplicate LinkedIn job postings
  • SQLite-backed persistence for every opportunity
  • Status and notes tracking per saved job
  • Dedicated Streamlit UI page
📡 Watch Engine
  • Collect articles from RSS feeds and scraped sources
  • Summarize technical content with LLMs
  • Suggest LinkedIn posts from watch articles

🏗️ Architecture

The repo is in active transition from a working prototype to a clean layered architecture. The target design separates business logic from runtime adapters and external integrations.

app/
Runtime adapters — Streamlit UI, FastAPI, CLI, background workers
domain/
Pure business rules, entities, statuses, use cases
services/
External adapters — LinkedIn, Medium, RSS, LLM providers
storage/
SQLite, migrations, repositories (replacing legacy JSON)
observability/
Structured logs, workflow run IDs, execution events

Every workflow carries a run_id. Every sensitive action leaves a persistent trace. SQLite is the source of truth — JSON files are transitional legacy being progressively migrated.
